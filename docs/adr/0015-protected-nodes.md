# Protected nodes: the plugin declares, the core enforces

**The split.** Some nodes are load-bearing structure a plugin can't let the user destroy — the daily plugin's "Daily" container guards every day note (a `removeNode` cascades the whole subtree). The seam draws one line: a **plugin decides _which_ nodes are protected** (`protects(nodeId)`, plugins/types.ts); the **core decides what "protected" _means_** and enforces it (`src/components/protection.tsx`). The plugin never re-implements a rule; the core never invents which nodes qualify.

**What protection blocks.** A protected node can't be **deleted**, **blanked** (its name heals on blur), **turned into a to-do**, or **marked completed**. It stays freely editable, renamable, and can take children — protection guards the node's existence and kind, not its content. The four rules are enforced at the core's own command funnels, so every path is covered at once: `onDeleteNode`, `onToggleCompleted`, `onSetTask` route through `guardProtected(id, kind, rowEl)`, and the blur heal (`protected-text.ts`) routes through `signalRejection`. Un-completing and un-tasking are always allowed (you're _removing_ a state, not imposing one on structure).

**`true` is a complete declaration.** `protects` returns `false | true | NodeProtection`. A bare `true` is enough: the core supplies generic rejection copy (`DEFAULT_REASON` in protection.tsx), so a protected node is a real, legible block — never a silent no-op — with zero plugin effort. The `NodeProtection` descriptor is **all overrides**: `reason` (general "why"), `blankReason`/`taskReason`/`completeReason` (per-action), and `canonicalText` (the name restored on blank). Resolution is per-action-override → `reason` → core default. The core does not depend on the plugin filling any of it in.

**Why a uniform reject, not a per-rule no-op.** A blocked action that just does nothing reads as a dropped keystroke. Every rejection gives the same feedback: shake the row (`rejectRow`, the "no" gesture, with a reduced-motion tint fallback) and toast the reason (keyed `protected-<kind>` so a rapid repeat replaces rather than stacks). One helper owns the shake+toast for all four kinds, so feedback can't drift between them.

**The lock is reactive.** A protected node wears an always-on lock signifier (`<ProtectedLock>`, rendered on the list row _and_ the zoomed title). Whether a node is protected can resolve **after** first render — the daily container's protection depends on the daily-index kv GET. So the render reads `useIsProtected(nodeId)`, which subscribes via the plugin's `protectsChanged` and re-evaluates when that async state arrives; a render-time `isProtected()` read showed the lock late (only after an unrelated re-render). A plugin whose protection is static (decidable synchronously) provides no `protectsChanged` and never perturbs the hook.

**Don't:** re-implement a protection rule inside a plugin (return a descriptor; the core enforces); add a fifth blocked action by sprinkling a new guard at its call site (add a `ProtectionKind` and route through `guardProtected`/`signalRejection`, so the lock, copy, and feedback stay in lockstep); make the core require descriptor fields (a bare `true` must keep working); or read `isProtected()` in render instead of `useIsProtected()` (the lock will lag async protection).
