# Daily notes nest in a calendar hierarchy: Daily > Year > Month > Week > Day

Status: accepted (2026-07-16)

Daily notes stop piling flat under the protected "Daily" container and nest as `Daily > YYYY > Month > Week > Day` (issue [#271](https://github.com/cameronapak/dotflowy/issues/271); grew out of an alpha-user ask for weekly "sprint" grouping, matching Workflowy's calendar structure). The container stays the root — no rename, the `container` sentinel in the daily-index kv is untouched. This amends [ADR 0041](./0041-daily-note-entry-point-seeding.md)'s implicit flat-container picture; 0041's seeding rules themselves are unchanged.

- **Weeks are ISO 8601 and atomic.** Monday start, ISO week numbers. A week never splits across months: its **Thursday** decides which month AND year own it — the ISO-consistent rule, which also equals majority-of-days (e.g. the 2026-06-29..07-05 straddle week is `2026-W27`, owned whole by July; `2025-12-29` belongs to `2026-W01` under January 2026). There is **no week-start setting in v1** — week identity is baked into persisted keys, so changing it later is a migration; the spec is fixed from day one.
- **Identity: bare, shape-disambiguated keys in the existing daily-index kv**, beside the `container` sentinel: `2026` (year), `2026-07` (month), `2026-W29` (week), `2026-07-16` (day). No prefixes — the four shapes are mutually unambiguous, and [ADR 0038](./0038-date-token.md)'s "the token interior IS the index key" rule then extends to every level for free (the period-token follow-up depends on exactly this shape). The pure math (ISO week number, week-year, Thursday rule, parent-key walk, same-kind comparator, labels) lives dependency-free in `src/data/date-links.ts` next to `localDateKey`, imported by BOTH the client and the Worker so the two placements can't drift.
- **Ordering flips to chronological ascending at every level** (years, months, weeks, days), replacing newest-first head-insertion inside the Daily subtree. Sorted insertion compares sibling date keys via the index reverse map. This retires the known "out-of-order past day lands on top" caveat.
- **Lazy scaffold, seed-free (ADR 0041 discipline).** Year/month/week nodes are minted only when the first day inside them is created, through the same atomic get-or-create cascade as days (`claimMapping` client-side, `getOrCreateKv` in the DO — one claim per level, so concurrent clients/agents can't mint duplicates). Scaffold nodes are pure structure with canonical text ("2026" / "July" / "Week 29"), never seeded content; `seedEntryLine` stays day-only at the three write-intent surfaces.
- **Every scaffold node is protected** (the daily container's four rules — no delete / blank / to-do / complete — with per-node `canonicalText`), on both sides of the trust boundary. Non-negotiable: `removeNode` cascades, so an unprotected week delete would take its days with it. Deliberate consequence: outdenting a bullet from under a day makes it a child of the week — moves under protected nodes are already legal (Send to Today does it), so the "promote a note to week level" ask costs zero new code.
- **The Worker is a lockstep second implementation, on purpose.** `planEnsureDaily` (`worker/outline-ops.ts`) and the `claimDailyId` cascade (`worker/mcp-tools.ts`) grow the identical chain + sorted insertion, and the server-side protection guard generalizes from the container to everything the daily-index maps. MCP tools are externally unchanged. This is the standing ADR 0026/0028 trade: agent writes land as one atomic DO batch, priced at maintaining placement logic twice — both sides import the same `date-links.ts` math so only the wiring is duplicated, and `worker/outline-ops.test.ts` covers the Worker copy (e2e can't reach it).
- **Migration is one-time, client-side, automatic** on first daily-note touch after ship — no prompt, because a declined prompt leaves a permanently half-structured container. It is fully derivable from the daily-index (every day key→nodeId is known): build the scaffold, `moveManyNodes` days into their weeks, reorder ascending — one `runStructural` batch (one Cmd+Z), streamed via `runStructuralSliced` over 500 ops, toast on completion. Idempotent: it keys off flat-shape day children actually present under the container, so re-entry after a partial run just finishes the job. **Scope is the scaffold only:** a mapped day is migrated only while it still lives inside the Daily subtree (under the container or a year/month/week node); a day the user has relocated elsewhere (dragged under a normal bullet) is left where it is, never dragged back. The Worker never migrates; a new MCP-created day landing in the new hierarchy while old flat days await the client is a fine intermediate state.

Non-issues checked at design time: the free-tier node ceiling (~65 scaffold nodes/year — negligible); date-chip clicks and `goToDate` (same get-or-create cascade, no UX change); quick-add Seam L / Send to Today / Mirror to Today (land under the day as before — only the day's placement moved).

Rejected: **prefixed scaffold keys** (`week:2026-W29`) — the bare shapes are already unambiguous and prefixes would break the ADR 0038 token-interior-is-key extension. Rejected: **configurable week start** — persisted week identity would then vary per setting, turning a preference toggle into a data migration. Rejected: **splitting straddle weeks per month** — Workflowy's own calendar keeps weeks atomic, and a split week has no single identity key. Rejected: **prompt-gated or server-side migration** — a declined prompt strands the container half-structured; the DO can't run TanStack-side move semantics and doesn't need to. Rejected: **renaming the container or re-keying the sentinel** — pure churn on a protected, referenced node.

Out of scope (v1): week-start display setting, "create my whole year" templates, custom recurring periods. Follow-up (separate issue, blocked by this one): period tokens `[[2026]]` / `[[2026-07]]` / `[[2026-W29]]` — this ADR only guarantees the key shape they need.
