# OPML import and export

Status: accepted (2026-07-07)

dotflowy imports real Workflowy OPML exports with **nothing silently lost** and exports OPML that **Workflowy itself imports cleanly**, on two surfaces: the app UI (More menu / Cmd+K) and an MCP tool pair. The fidelity bar is *degraded, never silent*: content always survives byte-exact; presentation may shift, and every shift is counted and disclosed. The whole decision set was worked as wayfinder map [#108](https://github.com/cameronapak/dotflowy/issues/108); evidence lives in `docs/spec-assets/opml/` (dialect reference, inline mapping table, parser comparison, fidelity-probe report) and was validated against a real 16,882-node / 1.95 MB Workflowy export — zero content-loss violations, ~131 counted degradations (<1%).

## The Workflowy dialect ([#112](https://github.com/cameronapak/dotflowy/issues/112))

A live paste-import → re-export round-trip is **byte-identical**, so "what Workflowy accepts" collapses into "exactly what it emits": `<outline>` with exactly three attributes — `text`, `_complete="true"` (present-iff-true), `_note` (newlines as `&#10;`). No ids, no timestamps. Inline rich text is **HTML escaped inside the attribute values**, which forces a two-layer decode, strictly ordered: XML-decode, then HTML-entity-decode (a user-typed literal `<` arrives as `&amp;lt;` and must survive as content). Export mirrors in reverse. The HTML scanner must be **tolerant**: real Workflowy exports contain malformed cross-bullet `<b>` spans (probe-verified), so strict HTML parsing would reject real files. Serializer cosmetics (indent, attribute order) are not load-bearing. Dynalist shares the `_note` convention, so honoring it covers more than Workflowy.

## Parser: `@rgrove/parse-xml`, one shared implementation ([#110](https://github.com/cameronapak/dotflowy/issues/110))

`@rgrove/parse-xml` v4 (4 KB gz, zero deps) is the one XML parser for both runtimes — SPA and Worker. It was the only small candidate correct on the actual hazards: it decodes `&#10;` inside attributes (fast-xml-parser corrupts every multi-line note), **fails loudly** on truncation with line/column (htmlparser2 and txml silently return a partial tree — the forbidden failure mode), and is entity-bomb/XXE-proof by construction. 29 ms on the full 17k-node export. The serializer is hand-rolled (~30 lines: escape `& < > " '`, newlines as `&#10;` in attributes) — output shape is fully under our control. Parse failures become a `Data.TaggedError` (`OpmlParseError` carrying line/col), the `worker/wire.ts` convention. Guard raw input size before parsing. Rejected: DOMParser-in-browser + library-in-Worker (two implementations of the fidelity-critical path will drift; the browser half can't run under `bun test`), saxes (maintenance mode, 3× adapter code), fast-xml-parser (broken attribute decoding), lenient parsers (never error — precisely wrong here).

## Notes: `_note` degrades to child bullets, no schema change ([#113](https://github.com/cameronapak/dotflowy/issues/113))

There is **no `note` Node field** — an importer doesn't get to drive the schema, and a field would be invisible data until the editor grew a whole notes UI. A bullet's `_note` becomes plain child bullets **prepended before its existing children**, one per line, order preserved, no marker; blank lines and the trailing serializer `&#10;` are dropped (formatting, not content). Export never emits or reconstructs `_note` — note-ness is lost on first import, by design (export-side re-interpretation of text conventions is fragile: a hand-typed marker would fake a note in Workflowy). The conversion is a counted, disclosed degradation. Rejected: a `note` field (schema + wire + DO + editor blast radius for day-one-invisible data), marker/tag conventions (visual noise, collision with user text), one-bullet-per-note (mangles or invents structure).

## Inline rich text: mapped, degraded-with-disclosure, never escaped ([#114](https://github.com/cameronapak/dotflowy/issues/114))

Full table in `docs/spec-assets/opml/inline-rich-text-mapping.md`. **No escape layer**: imported plain text behaves exactly as if typed in dotflowy (`*word*` arrives byte-intact and renders italic) — dotflowy has no escape syntax and inventing one fights the parsed-from-text model. Tags map to the shipped tokens (`<b>`→`**`, `<i>`→`*`, `<u>`→`~`, `<s>`→`~~`, `<code>`→backticks, `<a>`→`[label](url)`). Counted degradations: marker char inside a run drops formatting and keeps text; nesting flattens to outermost-wins with links trumping all; `<mark c-*/bc-*>` maps to `==highlight==` nearest palette color (depends on the highlights token, ADR 0035; fallback plain + disclose); `<mention id>` becomes an `@mention(id)` placeholder (the display name is not exported — unrecoverable); unknown tags strip to inner text (the forward-compat catch-all). `<time>` — ~16% of real nodes — is **adopted, not degraded**: it maps to the date token (ADR 0038), keyed on the canonical attrs, and export rebuilds `<time start*>` — a true round-trip. Export is the exact inverse of the import table, plus projections for dotflowy-only constructs: node links → `<a href="https://app.dotflowy.com/<id>">` with flattened target text, Bible refs → route.bible links, `#tags` pass through as plain text, highlight color emoji becomes the `bc-*` class.

## Import execution: client commit, one transaction, chunked changelog ([#111](https://github.com/cameronapak/dotflowy/issues/111), [#115](https://github.com/cameronapak/dotflowy/issues/115))

- **A shared pure core in `src/data`** maps OPML → forest → ops (the `wire-schema.ts` shared-leaf precedent, import direction). The app UI and the MCP tool consume the same core — "both surfaces" share a *core*, not an endpoint — so they cannot drift on mapping or counting.
- **The app commits client-side**: ONE `runStructural` batch through the existing `POST /api/nodes {ops}` → `applyBatch`. No new endpoint; ADR 0009's invariant, optimistic UI, echo-hold, and single-capture undo all fall out naturally. Rejected: a Worker-side import endpoint (new wire surface, breaks the undo constraint, duplicates the write path).
- **The binding constraint was never the 500-node tool cap but the 2 MB SQLite row cap**: `recordChange` stores a frame's ops as one blob, and a 17k-node import is ~4–7 MB of ops. Fix: `recordChange` splits large batches into **≤500-op changelog rows, each with its own seq, inside the caller's single `transactionSync`**; `broadcastChange` emits the N frames only after commit; `applyBatch` returns the final seq. Atomicity comes from the one transaction — any fault means **"nothing was imported"**, retry-safe. Emission is depth-first pre-order, so every frame prefix is chain-valid for live remote viewers. The chunking benefits every future big batch (including undo-of-import) with zero caller changes.
- **UX** ([#111](https://github.com/cameronapak/dotflowy/issues/111)): entry = More menu + Cmd+K ("Import OPML…", the ADR 0034 bridge) opening a hidden file input; parse client-side (~29 ms), then one dialog carries the whole flow — summary/confirm **with explicit degradation disclosure**, modal progress, success ("Go to imported") or error. Every import lands in one fresh top-level container (`Imported from Workflowy — {date}`), **always-append** (re-import = second container, no dedup), created `collapsed: true` (perf guard: `buildVisibleRows` never descends into a collapsed root, so 17k optimistic inserts can't thrash). Undo = one history `capture` before commit; one Cmd+Z removes the whole import. Drag-drop deferred.
- **Ceilings**: app import 50,000 post-split nodes (counted after note line-splitting), enforced friendly in the dialog and hard at the Worker decode. `MAX_BATCH_NODES = 500` stays untouched for `add_subtree` (an ergonomics bound, ADR 0028, not a storage limit).

## Export semantics: content, never state ([#116](https://github.com/cameronapak/dotflowy/issues/116))

Scope is zoom-root-or-all (the markdown precedent), root included. Carried: text, structure, order, `completed` → `_complete`, `isTask` → an invented `_task="true"` (task-ness is content; Workflowy shows a plain bullet — disclosed). Dropped: `collapsed` (view state), `bookmarkedAt`, `origin` (provenance is stamped by the importing side — carrying it would lie), timestamps (OPML is interchange, not backup). **Mirrors persist via an invented dialect** — Workflowy has none to copy (its own exports lose mirror identity): emit the full resolved duplicate plus `id` on in-scope mirror sources and `_mirror="<sourceId>"` on mirror roots. dotflowy import re-links when the source is in-document, else a disclosed detached copy; `sourcesOnPath` caps cycles. Live-verified: Workflowy tolerates the extra attributes on paste (no fallback encoding needed) but **strips them on re-export** — mirror identity survives dotflowy→dotflowy only. File mechanics: `dotflowy-export-<date>.opml` (zoomed adds a slug), `text/x-opml;charset=utf-8`, via a new `downloadTextFile` helper; `<head>` is `<title>` only — **no `ownerEmail`** (privacy leak in a shareable file).

## MCP tools: programmatic, not migration ([#117](https://github.com/cameronapak/dotflowy/issues/117))

A multi-MB export can't travel as a tool argument (~500k tokens), so `import_opml` is for agent workflows and **full Workflowy migration is explicitly the app UI's job**. `import_opml` mirrors `add_subtree` targeting (`parentId` | `date` | top-level; **no synthetic wrapper container** — the fresh-container rule is app-UI presentation, not data semantics), has a 5,000 post-split ceiling counted by the same shared-core planner, returns a **compact receipt** (root ids + texts, counts, the degradation tally — the only place MCP disclosure can live), and takes `dryRun` (plans, commits nothing — the MCP twin of the confirm step). `export_opml` mirrors `get_outline` scope and returns the raw OPML string with no preamble; same ceiling, **reject-never-truncate** (truncated OPML that still parses is silent loss). Imported nodes get the OAuth client's `origin` stamp — provenance tracks the door, not authorship. No URL-fetch argument (an authenticated SSRF surface re-treading `unfurl.ts` for territory already covered). Both tools are thin shells over the shared core; `import_opml` requires the chunked `recordChange`.

## Out of scope

Workflowy's other export formats (plain text, JSON backup); live/continuous Workflowy sync — this is one-shot file interchange. Generic well-formed OPML imports best-effort by construction (the parser is dialect-agnostic; `text`/`_note`/`_complete` covers Dynalist too), but first-class treatment of other dialects (Dynalist, OmniOutliner, Logseq) is deliberately unresearched — a future effort if wanted.
