# A query grammar for the view filter, with plugin-registered operators

Filtering by tag exists (`?q=#a #b`, click-driven, AND-only) but nothing else is filterable — no node kind, no completion state, no text — even though every axis is already derivable from `node.text` plus existing fields. We decided to grow `?q=` into a real query grammar rather than build a search feature, and to split the grammar between a core parse engine and plugin-registered operators. Research base: [docs/research/workflowy-search-operators.md](../research/workflowy-search-operators.md).

## The decision, in nine parts

1. **Filter, not search.** Dotflowy keeps its existing split: Cmd+K is navigation (fuzzy jump, ADR 0034), `?q=` is filtering (a render-time view transform — URL-driven, zoom-scoped, shareable, never mutates `collapsed`). The grammar extends the filter. We rejected a Workflowy-style unified search box: it would fuse two things this codebase deliberately factored apart, and the `?q=` + Seam G architecture already _is_ Workflowy's filter semantics (live, ancestor context, zoom-scoped) — it was merely illiterate.

2. **Core engine, plugin meaning.** The core owns the tokenizer/parser — spaces (AND), `-` (NOT), `OR`, `"quotes"`, and the `key:value` shape — as a pure, dependency-free module in `src/data/` (the `tags.ts`/`highlight.ts` discipline: no DOM, no collection imports, bun-testable). A new **`filterOperators` seam on `definePlugin`** lets each plugin register the operators it owns semantically: a key (or term shape), its values, a one-line description, and a `(node, index) => boolean` predicate. Collisions are guarded at load like keymap keys. Core parses _structure_; plugins supply _meaning_. Core itself registers the node-kind operators (`is:todo|bullet|paragraph` — `kind`/`isTask` are core fields, ADR 0045, render tie-break applies) the same way core commands sit beside plugin commands in the `/` palette. We rejected a closed core grammar (every new axis becomes a core PR; `is:complete` hard-codes todos knowledge into core) and rejected growing the tags plugin into a filter plugin (`is:todo` living in the tags folder is a lie that worsens every release). Consequence: the `?q=` Seam-G view transform migrates out of the tags plugin into core; tags keeps chip-click → add-`#tag`-term.

3. **Free text matches.** A bare term is a case-insensitive substring match over `flattenInline(node.text)` — what you see is what matches (folded links match their labels, and per ADR 0043 your own filter sees inside your own spoilers). `"quoted phrase"` is an exact phrase. **No fuzzy matching**: fuzzy is right for a jump list and wrong for a filter, where the view must feel deterministic; fuzzy stays Cmd+K's differentiator.

4. **v1 operators** (all zero-migration): `#tag` (tags), `is:todo`/`is:bullet`/`is:paragraph` (core), `is:complete` (todos), `is:mirror` (core — `mirrorOf`), `has:link` (links — the operator Workflowy's community requested for a decade and never got), `highlight:` bare + `highlight:red|orange|yellow|green|blue|purple` (highlight, its six-emoji palette), `is:agent` (provenance — `origin`, an axis Workflowy cannot offer), and `-` negation on any term.

5. **`OR` ships in v1**, with Workflowy's exact semantics: uppercase standalone `OR` binds the two adjacent terms; everything else ANDs. The model is an AND-list of groups, a group is one-or-more OR-ed terms — the term-predicate shape absorbs it (`some` within a group, `every` across groups). Shipping it now avoids a grammar-compat break later (once `?q=` is public, the bare word `OR` changing meaning would silently rewrite existing saved URLs). **No parentheses, ever, in v1** — that is where grammar complexity actually starts.

6. **The input is summoned, and stays while a query is active.** It lives in the subheader (contextual chrome, ADR 0002), opened by **Cmd+F**, a Cmd+K action row ("Filter this view"), or the **header magnifier** — which now summons the filter, not the switcher; a magnifier that navigated was a small lie, and the switcher keeps its touch entry via a new `Command` (⌘) header button beside it ("Command center", ADR 0034 language). Hijacking Cmd+F is a repair, not a theft: virtualization (ADR 0019) already broke native browser find — off-window rows aren't in the DOM. The input shows the raw query with live-as-you-type filtering (debounced, `router.replace` so history isn't spammed), a leading muted search icon, and a trailing clear X (shown when non-empty; clears text + `?q=`, keeps focus). **While a query is active the input stays resident** — _amended 2026-07-11, superseding the original pill design:_ the blurred-with-pills state was the tag-era paradigm, and once terms include free text and operators the raw string is the honest representation; per-term pill removal lost its value (clear-and-retype is cheaper than parsing which pill to X). Empty text + blur collapses the row. Enter commits (flushes the debounce) and blurs to the outline. Escape inside the input is progressively "less search": close the suggestion popover → clear the text (focus kept) → collapse the row; the window-level Escape (caret in the outline, filter active) still clears in one press, with the existing node-text-focused and `[role="dialog"]` guards. _Amended 2026-07-13:_ the **magnifier is a toggle, not open-only** — a second press dismisses (clears `?q=` + collapses); `preventDefault` on its pointerdown keeps the input focused across the press so a blur doesn't collapse-then-reopen before the click lands. It **lights solid** (`variant="default"`, ADR 0033's grayscale "on" idiom shared with `SpotlightIndicator`) and carries `aria-pressed` while a query is active, so the destructive toggle-off press only fires while the control visibly reads as "on". This toggle-and-clear behavior is the **magnifier only**; Cmd+F and the Cmd+K action stay open-only (a keyboard people spam shouldn't wipe a filter on a repeat press). The suggestion popover, portaled under the input, defers its reveal until the subheader's expand animation settles (`SUBHEADER_EXPAND_MS`) — opening it mid-slide pins it to a stale, fixed input rect — then eases in via a dedicated CSS keyframe (reduced-motion disables it).

7. **Registry-driven autocomplete.** The operator metadata that plugins register also generates the input's suggestions: empty-focus shows the operator cheat sheet, `key:` shows its values, `#` shows the tag corpus (already maintained in `TreeIndex.tagCorpus`), `highlight:` shows color swatches. Nothing hand-maintained — a new plugin operator appears in autocomplete for free. This is the deliberate over-investment: Workflowy's operators went unused for a decade until its search bar grew a suggestion widget; hidden grammar is dead grammar. Scope guard: suggestions only insert text into a plain `<input>` — no inline pill editing while typing, no Seam-H engine (that's contentEditable-specific), no cmdk.

8. **A match reveals its subtree.** Previously a match's non-matching children were pruned entirely — filter to a tagged project and you got a dead end. Now a match's descendants render under normal visibility rules (collapse respected, chevrons work, hide-completed still applies), undimmed; ancestors stay dimmed context. This honors the never-mutate-`collapsed` invariant — visibility ≠ collapse state — and matches Workflowy's expand-while-searching.

9. **Explicit non-goals**, each with a reason, not just scope-trimming:
   - ~~**Saved searches**~~ — _promoted 2026-07-11:_ designed in [ADR 0048](./0048-saved-filter-queries.md).
   - **MCP parity** — `search_nodes` stays substring-only; the parser's purity is the recorded constraint that makes "Worker imports the grammar like it imports `tree.ts`" a clean follow-up, after its spoiler-redaction interaction (ADR 0043) gets its own pass.
   - **`created:`/`changed:`** — the only axis needing Node timestamps (wire-schema + DO migration + fixtures + `withNodeDefaults`); its own ADR when wanted.
   - **`A > B` nested search** — real parser complexity; zoom-scoping covers the common case.
   - **Date search, `text:bold`, `is:backlinks`** — natural-language date parsing wants design time; nobody filters by italic; "is a backlink" vs "has backlinks" is ambiguous.

## Consequences

- Text edits do not live-re-prune the filter: the visible-rows list rebuilds on `structureRev` changes only (ADR 0019), so a node edited to no longer match stays visible until the next structural change. Kept deliberately — the node you're editing must not vanish under your caret, and this is today's tag-filter behavior.
- `?q=` becomes public grammar. Every syntax decision here is one-way once users share URLs; that is why `OR` and quoting ship day one rather than being retrofitted.
- The filter must stay exposed through `getViewFilter()` (view-state.ts) for event-time reads — structural paste's hidden-landing rule (ADR 0044) depends on it.
- _Amended 2026-07-12:_ **caret/focus walks share the render's filtered rows.** `findVisibleNeighbor` and `lastVisibleDescendant` take the active `QueryFilter`, and every event-time caller (arrow nav, delete-focus, selection caret exits) passes `getViewFilter()` — reversing the original "caret nav walks the unfiltered visible tree" call, which made arrows silently no-op while filtering (the computed neighbor was a row the filter had pruned out of the DOM). Nav covers **all** rendered rows: matches, revealed descendants, and dimmed ancestor context (they're editable; the caret lands wherever the eye sees a row). `focusKeyFor` deliberately stays unfiltered — it runs pre-re-render inside `runStructural` (the filter is stale) and derives a key from candidates rather than a sequence, so a superset walk is strictly safer. Drag-while-filtered and Shift+arrow selection over hidden siblings are tracked separately.
