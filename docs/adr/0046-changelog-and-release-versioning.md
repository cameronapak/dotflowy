# The changelog: semver as disclosure, fragments as source

Status: accepted

Dotflowy publishes a **user-facing changelog**, versioned with **semver**, authored as
**per-PR fragments** ([changesets](https://github.com/changesets/changesets)) that CI refuses to
let you skip. The public record is **GitHub Releases**; the in-app surface is a **dialog with an
unread badge**. Nothing is written twice: one fragment set produces both.

## MAJOR means "a reader has to do something"

**Nobody can pin Dotflowy.** Humans get whatever the last `bun run deploy` pushed. Agents
negotiate MCP's `protocolVersion` (the spec date), not our `serverInfo.version` — no MCP client
pins to a server version, and there is no `^1.2.0` anywhere in the system.

So semver here is **communicative, not contractual**. It is a signal, not a guarantee, and the
bump levels are defined by _disclosure_, not compatibility:

- **MAJOR** — a reader must take an action: relearn a gesture, fix an agent prompt, re-export a
  file, update a tool call.
- **MINOR** — a new capability; nothing to do.
- **PATCH** — it just got better.

This is the only rule that survives "nobody can pin," and it is checkable at PR time with one
question. Compatibility-flavored rules ("MAJOR = the wire schema changed") were rejected: they
promise a stable version to stay on, and no such version exists.

**A bump nobody trusts is worse than no version at all** — the same failure mode as a badge that
cries wolf. Two mechanisms defend the rule: CI (`changeset status --since=main`) makes the
disclosure mandatory, and the version reaches running tabs (below) so MAJOR has a delivery path.

## A fragment is `{bump, summary}` — there is no category

The bump type **is** the reader's axis. MAJOR/MINOR/PATCH already say Changed-with-action / Added
/ Fixed, with better semantics and a CI gate behind them. A second `category` taxonomy would need
defining, validating, rendering, and keeping honest in order to say what the first one already
says. Don't add one.

## The fragments are the source; `CHANGELOG.md` and the app's data are siblings

`changeset version` **consumes** the fragments. So `bun run release` **archives them first**, into
`changelog/<version>/`, alongside one appended `{version, date}` manifest line. Then:

- `CHANGELOG.md` (generated by changesets) is the **input to the GitHub Release**, verbatim. That
  is who reads it. Polish its prose freely.
- `changelog/**` (the archived fragments + manifest) is the **app's data**, parsed and validated
  with Effect Schema at **build time**, compiled into the bundle. A malformed entry fails the
  build, not production.

**Neither is parsed from the other.** Parsing `CHANGELOG.md` into structured data was rejected: it
makes the product's data model a hostage to a markdown format we don't own and will hand-edit.

**Skipping the archive is impossible, not merely discouraged.** `bun run build` fails when
`changelog/<package.json version>/` is missing. A convention loses to a tired human at 11pm and to
an agent that reasoned its way to `changeset version`; a build-time invariant does not. Same spine
as [ADR 0014](./0014-validate-the-worker-do-trust-boundary.md) — validate at the boundary, fail
closed.

The release script also emits `changelog.json` as a public static asset, so a branded page later
(on either domain) is a pure renderer rather than a rewrite.

## The cursor is `lastSeenVersion`. There is no `seq`

Releases live in a build-time array, newest first. **Array order is release order.** "Is there
news" is `releases[0].version !== lastSeenVersion`; "how many did I miss" is a `findIndex`. No
integer compare, no semver parser in the client bundle.

A monotonic `seq` was proposed and rejected: with semver present it is **redundant state that can
desync**, and two identities for one release is a bug waiting for a bad merge.

The cursor is **one row in a `/api/kv` side-collection** ([ADR 0008](./0008-sync-via-a-per-user-durable-object.md)),
so it syncs across a user's devices. It passes its **concrete** Effect Schema inline — ADR 0008
forbids a generic `createKvCollection<T>` factory.

**Read it with `subscribeChanges` + `useSyncExternalStore`, never `useLiveQuery`**, which hard-fails
the `/` prerender (see `src/data/tag-colors.ts`). The unread signal rides the header's "More" menu
(a quiet notification dot on the trigger, and emphasis on the "What's new" item when the menu opens
— not a standalone header CTA, which read as too loud), and the header renders during that prerender.

**`getServerSnapshot` returns `null`, and `null` is ambiguous** — it means both "the collection
hasn't loaded" and "this user has no cursor row." Distinguish them via the collection's `status`,
or ship one of two bugs:

| state             | badge                                          |
| ----------------- | ---------------------------------------------- |
| not ready         | hidden                                         |
| ready, **no row** | hidden — **seed `lastSeen = latest` silently** |
| ready, `= latest` | hidden                                         |
| ready, `< latest` | shown                                          |

**Seeding silently is the load-bearing choice.** The first badge a user ever sees defines what the
badge means. Firing it for a release they already lived through teaches them, on day one, that it
is not about _change_. Announce the changelog's existence some other way; don't borrow the badge's
credibility to do marketing — that loan can never be repaid.

## The in-app surface is a dialog in `__root.tsx`

Mounted once, beside `McpConnectDialog` / `OpmlImportDialog` / `DeleteConfirmDialog`, behind a
module-singleton opener (the `move-dialog-opener.ts` pattern). Entry points: the header More menu
(its "What's new" item, emphasized while unread) and a Cmd+K global action. MAJOR renders a
"Breaking" callout derived from the bump type already in the data — not a louder badge, and **no new
state**.

It is **not** the Tier-3 panel: `openPanel` is the containment boundary
([ADR 0031](./0031-two-lane-plugin-trust.md)) for plugin and Lane-B UI, and core chrome must not
squat in the room built to hold untrusted code. It is **not** a route: SPA mode
([ADR 0008](./0008-sync-via-a-per-user-durable-object.md)) prerenders the _shell_, so a
`/changelog` route is neither crawlable nor shareable — and everything under `__root`'s `AuthGate`
is authed-only anyway, which is correct for a badge and wrong for a public page.

## Running tabs learn the version from the sync handshake

**A changelog cannot inform a stale client, and stale clients are exactly who MAJOR is for.** An
outliner tab stays open for days; its release array is frozen in the bundle it loaded. Worse, that
tab talks to today's Worker, so a moved wire shape gets a `400` at the trust boundary and the user
experiences "my edits stopped saving," never "I should reload."

So `serverVersion` (compiled from `package.json` into both bundles via Vite `define`) rides the
`snapshot`/`resume` frames the DO already sends on every connect. The client compares it to its own
compiled-in version; a mismatch offers a non-blocking reload.

Verified: Effect Schema's `decodeUnknownExit` **ignores excess properties**, so old clients decode
the new field and drop it. Adding it cannot break a tab mid-session.

**This is not the changelog.** It is a distinct mechanism, built in its own PR, recorded here
because MAJOR is unenforceable without it — otherwise "you have to do something" is a sentence in a
file the affected user will never load. It also forces the fix for `SERVER_INFO.version`
(`worker/mcp.ts`), today a hardcoded `"0.1.0"` duplicated from `package.json` and stale by
construction: adopting semver without it ships `2.0.0` to humans and `0.1.0` to every agent on the
same deploy.

## The public changelog is GitHub Releases

The repo is public. `gh release create` in the release script gives, for zero build code: a
crawlable page, permanent per-version URLs, immutable git tags binding each release to a commit,
and an **Atom feed** at `/releases.atom` — the entire RSS requirement.

The cost is accepted: a code host is a jarring destination for an outliner's users. The in-app
dialog is the primary surface; the public page serves links, feeds, and crawlers, and those three
don't care about brand.

## Considered and rejected

- **Changesets' default flow, where every fragment ships verbatim.** Rejected in spirit but adopted
  in fact — with the escape hatch that a PR simply _doesn't add a fragment_ when it isn't news
  (`chore:` commits, agent skill locks). The folder is an inbox curated at PR time, not a manifest
  of every merge.
- **A monotonic integer `seq` as release identity + cursor.** Redundant once semver exists.
- **A `category` field alongside the bump type.** Says what the bump already says.
- **`localStorage` for the cursor.** Rejected: "have I read this" is account state, not device
  state, and the kv collection is ~35 lines (`tag-colors.ts`).
- **Parsing `CHANGELOG.md` at build time.** Product data hostage to prose formatting.
- **The changesets GitHub Action / Release PR.** Real enforcement, but stands up a release pipeline
  that doesn't exist (`ci.yml` is quality-only) for a solo repo. The build-time invariant enforces
  the same thing for free.
- **A branded page on the landing site, fed by the app build.** Cross-repo coupling with a
  deploy-ordering hazard: the changelog goes stale until the _other_ site rebuilds. `changelog.json`
  keeps this reversible.

## Don't

Reintroduce a `seq`; add a `category`; parse `CHANGELOG.md` for app data; run `changeset version`
outside `bun run release` (the archive is silently lost, and the app renders a version with no
entries); reach for `useLiveQuery` in the badge (prerender dies); treat a `null` cursor as "unseen"
(the badge flashes, or every existing user is badged on ship day); render the changelog into
`openPanel`; or leave `SERVER_INFO.version` hardcoded.
